## 0. 문서의 위상

이 문서는 wiiiv에서 **Spec의 정체성·목적·한계**를 규정한다.

- **Spec은 실행 명세가 아니다**
- **Spec은 판단을 가능하게 하는 재료다**
- **Spec은 Blueprint 이전 단계의 자산이다**

> 이 문서를 벗어나는 Spec 정의는 **확장이 아니라 설계 위반**이다.

---

## 1. Spec의 정체성

wiiiv에서의 Spec은 실행 명세나 구현 가이드가 아니다.

Spec은 **하나의 실행이 결정되기 이전에, 그 판단을 가능하게 만든 모든 사고의 재료(raw material)**를 담는 자산이다.

- Spec은 결과물이 아니라 **판단 이전의 기록물**이다
- Spec은 요약이나 단순화를 목표로 하지 않는다

---

## 2. Spec의 목적

Spec의 목적은 다음 하나로 수렴한다:

> **Governor 혹은 DACS가 "이 실행을 해도 되는가, 그리고 무엇을 실행해야 하는가"를 판단할 수 있을 만큼 충분한 재료를 제공하는 것**

이를 위해 Spec은 다음을 가능한 한 풍부하게 보존한다:

| 항목 | 설명 |
|------|------|
| 사용자의 실제 요구 | 무엇을 원하는가 |
| 대화의 흐름 | 어떤 맥락에서 나온 요구인가 |
| 요구의 변경, 망설임, 강조 | 의도의 뉘앙스 |
| 명시적·암묵적 제약 | 해야 할 것과 하지 말아야 할 것 |
| 정리된 설명과 맥락 | 배경 지식, 도메인 정보 |

---

## 3. Spec은 무엇이 아닌가

Spec은 다음이 **아니다**:

```
❌ 실행 계획
❌ Blueprint
❌ Gate 입력
❌ Executor 입력
❌ 요약된 요구사항 목록
```

Spec은:
- 실행을 지시하지 않는다
- 실행 방법·순서·비용·권한을 정의하지 않는다

> **Spec은 "무엇을 해라"가 아니라 "이런 맥락에서 이런 요구가 있었다"를 담는다.**

---

## 4. Spec과 Blueprint의 관계

Spec은 Blueprint의 **입력 재료**다.

```
Spec
  → Governor 판단
  → (선택) DACS 합의
  → Execution Blueprint 생성
```

### Blueprint 생성 이후

- Spec은 더 이상 "살아있는 문서"가 아니다
- 판단 당시의 상태가 **고정된 스냅샷**으로 Blueprint 내부에 포함된다
- 이후 실행은 **Blueprint를 기준으로만** 이루어진다
- Spec은 **기록·재현·감사**를 위한 판단 자산으로 남는다

---

## 5. Spec의 형식 원칙

### 5.1 길이와 형식

Spec에는 **길이 제한이 없다**.

| 허용 | 설명 |
|------|------|
| 길어도 된다 | 맥락 보존이 우선 |
| 서술적이어도 된다 | bullet point 강제 아님 |
| 인간적인 표현 포함 가능 | 감정, 망설임, 강조 등 |
| 대화에 가까운 구조 허용 | 정형화 강제 아님 |

### 5.2 정제에 대한 원칙

Spec은 단순한 채팅 로그가 아니라, **판단을 위해 정제된 대화 기록**이어야 한다.

단, 다음 원칙을 따른다:

- 판단과 무관한 잡음(noise)을 **반드시** 보존할 의무는 없다
- 중복·번복·감정적 반복은 정리될 수 있다
- **정제는 선택적**이며, 강제되지 않는다

---

## 6. Spec의 최소 조건

Spec은 다음 조건만 만족하면 충분하다:

| 조건 | 설명 |
|------|------|
| 판단 가능성 | Governor가 독자적으로 판단할 수 있을 만큼 충분할 것 |
| 검토 가능성 | DACS에 위임했을 때 의미 있는 검토가 가능할 것 |
| 재현 가능성 | "왜 이 실행이 나왔는지"를 나중에 다시 읽어도 이해할 수 있을 것 |
| 비압축 원칙 | 요약·압축을 강제하지 않을 것 |
| 단계 명확성 | Blueprint 이전 단계의 자산임이 명확할 것 |

이 외의 형식적 요건은 강제하지 않는다.

---

## 7. 운영 가이드 (Operational Notes)

### 7.1 불충분한 Spec의 처리

```
Spec이 불충분하다고 판단될 경우,
Governor는 Blueprint를 생성하지 않고
추가 질문을 통해 Spec을 보완해야 한다.
```

- "충분함"의 최종 판단자는 **Governor**다
- Spec 작성자는 "완벽해야 할 의무"에서 벗어난다
- **불충분한 Spec은 오류가 아니라, 질문을 유발하는 상태**다

### 7.2 정제 책임

```
Spec의 1차 작성 책임은 요청자(또는 시스템)에 있다.
Spec의 정제는 선택적이며,
Governor는 필요 시 원본 대화 또는 추가 설명을 요청할 수 있다.
```

- "정제 안 하고 던지기" → 허용
- "과도한 정리" → Governor가 거부하고 재질문 가능
- 정제 실패가 실행으로 이어질 수 없다 (Blueprint는 Governor가 생성하므로)

### 7.3 길이 한계 대응

```
Spec 자체에는 길이 제한이 없다.
다만 Governor의 context 한계를 초과할 경우,
요약본과 원본을 함께 유지하는 구조를 사용할 수 있다.
이때 요약은 원본을 대체하지 않으며, 참조용으로만 사용된다.
```

- 요약은 허용된다
- 그러나 **요약은 대체물이 아니다**
- **원본 Spec은 항상 기준 자산**이다

---

## 8. 핵심 정의 (요약)

> **Spec은 실행을 위한 명령서가 아니라, 그 실행이 정당화되기까지의 사고와 맥락을 충분히 보존한 판단 자산이다.**

---

## 9. 설계 원칙 요약

```
Spec은 판단 자산이다
Spec은 실행을 지시하지 않는다
Spec은 Blueprint의 입력이다
Spec은 요약을 강제하지 않는다
불충분한 Spec은 질문을 유발한다
```

---

## 10. 최종 선언

**Spec은 wiiiv에서 가장 풍부한 자산이다.**

요약하면 손실되고,
압축하면 맥락이 사라진다.

Spec을 얇게 만드는 것은 효율이 아니라 **판단력의 훼손**이다.

Governor가 올바른 판단을 내리려면,
**충분한 재료**가 있어야 한다.

그 재료가 바로 Spec이다.
